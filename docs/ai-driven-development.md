# AI駆動開発の実践 — 初リリース作品の独学開発で実際にやったこと

「AIに丸投げ」でも「人間が全部理解して指揮する」でもない、
その間にある**実用的なワークフロー**を、本作の制作で確立しました。
誇張せず、実態のまま記載しています。

---

## 自分の前提条件

- IT学科の**学生**
- ゲーム業界経験ゼロ・Unity独学
- **本作が初リリース作品**
- **プロのような正確な仕様指示を出せる経験値がなかった**（独学出身として、要件を厳密な日本語で書き下す訓練を積んでいない）

この前提から出発したため、後述するワークフローは
「**仕様書を厳密に書ききれない独学者が、AIに頼って制作時間を最優先する**」
という制約条件下で最適化された運用になっている。

---

## 採用した運用フロー（実態）

開発工程を**4ステップ**で運用する。

### ステップ1：大仕様 → 小タスクへの分解

1. 大きな仕様を自然言語で AI に渡す（例：「環境プリセット保存/復元機能、24項目、座標復元時はフェード」）
2. AI がそれを小さなタスクに分解する（例：データ構造定義 → 保存API → 復元API → UI雛形 → 各分野復元 …）

人間が頭の中で粒度を決めるのではなく、**仕様の咀嚼と分解そのものを AI に任せる**。
独学者にとって「タスク粒度をどう切るか」自体が難しい判断なので、
AI に提案させて取捨選択する方が速い。

### ステップ2：各タスクの具体的な実装計画

ステップ1で分解した小タスクごとに、AI が具体的な実装計画を立てる。
例：保存APIなら「JsonUtility.ToJson で書き出し、
`Application.persistentDataPath/環境プリセット.json` に保存。Slot index は配列インデックス。
新フィールド追加時の互換性…」のレベルまで詳細化する。

計画粒度は「**弱いエンジンでも実装できる**」レベルまで詳細に。
粒度が荒いと、実装時の判断ブレ・手戻りが増える。

ポイント：

- **AI が AI に向けて指示書を書く**形になるため、後続セッションでも文脈が引き継げる
- 「**弱いエンジンでも実装できる粒度**」が目的（必ずしも安価モデルに渡すとは限らない。
  本作は実装も Opus 系で行ったが、計画の解像度を上げること自体が実装品質を上げる効果があった）

### ステップ3：実装ループ（PDCA）

各小タスクごとに、以下の PDCA ミニループを回す：

1. **Plan**：ステップ2の計画から1つの小タスクを選ぶ
2. **Do**：AI に実装させる（目的だけ伝える、自分で仕様書を厳密に書かない）
3. **Check**：VR で実プレイして違和感・バグを発見（修正してほしい箇所のスクショと要望程度で OK）
4. **Act**：違和感を AI に投げて修正させる → 再プレイ
5. 小タスクが固まるまで PDCA を回し、次の小タスクへ

このループの間、**人間はほとんどコードを読まない**。
デバッガーとして VR で動作確認し、違和感を言葉にして AI に投げる役割に徹する。
理由は[次節のボトルネック議論](#重要な気付き人間がコードを読む時間が最大のボトルネック)を参照。

### ステップ4：把握フェーズ（機能が固まってから）

1. 小タスク群すべて PDCA 完走
2. AI に「この機能の中身を順を追って説明して」と頼む
3. コードを後ろからなぞって読む（→ 「トップダウン×なぞり学習」）
4. 全体像と Unity の流儀が頭に入る

「**機能が固まるまでコードは読まない**」のは、修正のたびにコードが書き換わるため、
途中で読んでも次の修正で陳腐化するから。

詳細は[独学者向け学習論](#独学者向け学習論トップダウン--なぞり学習)を参照。

---

### 重要な気付き：「実装中は読まない、機能完成後に読む」のメリハリ

> 注意：これは「**バイブコーディング**（仕様も読まずAIに丸投げして
> 結果も読まない）」とは違う。**実装中はあえて読まずに PDCA を回し、
> 機能が固まってから読む**、という時系列の使い分け。

最初は「ちゃんと自前で理解しながら作る」つもりだった。が、すぐに気付いた：

- 1機能あたりのAI生成コード = 数百行
- それを毎回読んで把握する = 30分〜1時間
- 1日10機能実装すると = **読む時間だけで5〜10時間消費**

実態として、修正のたびにコードが書き換わるなら、その都度読み直すのは
**コストが重すぎる**。だから運用を以下のように切り替えた：

- **実装ループ中**：読まない。修正で陳腐化する、デバッガーとして VR で確認する方が価値が高い
- **機能が固まった後**：必ず読む。AI に説明させながら把握。把握しないと次機能に影響が出る
- **バグ調査で詰まったとき**：読む。違和感ベースで言葉を投げても解決しない場合は、コードを読んで具体的な指示を投げ直す

つまり「コードを読まない」のは**期間限定**で、フローの一部。
「丸投げで結果も読まない」のとは明確に違う。

### 自分のメイン役割は「デバッガー」

ゲームは**正しく動くことが至上**。コードが綺麗でも、VRヘッドセットを被って
プレイしてみて違和感があれば価値ゼロ。

逆に言えば、**動作確認は人間にしかできない**。AIにはVRで実プレイできない。
そのため自分のメイン役割を以下に定めた：

- VRで実プレイし、違和感を言葉にする
- 「想定の挙動」と「実際の挙動」を比較する
- AIに具体的なフィードバックを投げる

「**指揮者ではなくデバッガー**」というポジショニング。
ただし機能が固まれば**必ずコードを読んで把握**するため、最終的には
コードベースの全体像と Unity の流儀が頭に入っていく（後述の[なぞり学習](#独学者向け学習論トップダウン--なぞり学習)）。

## ツール変遷

### 初期：Google AI Studio（リポジトリ作成前）

開発開始時は **Google AI Studio**（Gemini）のチャットUIを使っていた。
**git リポジトリを作成する前の期間**で、この時に
[OutfitManager](../src/OutfitManager/README.md)（衣装動的着替え + 物理連携）系の
実装まで進めていた。

#### ボトルネック
- **直接ファイルを操作・閲覧できない**
- 新規チャットを起こすたびに、**必要なコードを全部手でコピペ**して貼る必要がある
- 大規模なリファクタや横断調査が事実上不可能
- コンテキストが切れるたびに同じ説明を繰り返す疲弊

このフェーズが**最も開発ペースが遅かった**。リポジトリ作成前の作業なので
git 履歴には残っていない。

### 移行：Claude Code（リポジトリ作成以降）

Google AI Studio で OutfitManager 系を作った後、Claude Code に完全移行して
**git リポジトリを作成し（2026-03-21）、それ以降の開発はすべて Claude Code 経由**で行った。

このフェーズで実装した機能群：
[ActionMatrix](../src/ActionMatrix/README.md), [AutonomyModule](../src/AutonomyModule/README.md),
[ExpressionCascade](../src/ExpressionCascade/README.md), [ExcelPipeline](../src/ExcelPipeline/README.md),
[DemoBuild](../src/DemoBuild/README.md), [PersistenceLayer](../src/PersistenceLayer/README.md),
[VRWorldSpaceUI](../src/VRWorldSpaceUI/README.md), [GameEventHub](../src/GameEventHub/README.md) 等。

Claude Code 移行で開発ペースが大きく上がり、そのおかげでリリースまで到達できた。

### 移行：Claude Code (Maxプラン)

主力AI開発ツールを Claude Code に移行。
- **Anthropic Claude（Opus系）** をメインモデル（プラン内で常に最新Opusを利用）
- ターミナル直結で Unity プロジェクトを直接操作
- ファイル読み書き / Bash 実行 / Grep / 並列 Sub-agent / WebFetch を統一インターフェースで提供
- カスタムコマンド・skills・hooks による拡張性

#### Claude Code の最強ポイント：YAML経由で何でも読める

Unity プロジェクトには `.unity` シーンファイル、`.prefab`、`.asset` (ScriptableObject) が
すべて **YAML テキスト** として保存されている。

Claude Code は **これらの YAML を直接読める**。これが何を意味するか：

- Inspector で設定した SerializeField の値を AI が確認できる
- シーン階層 / GameObject 名 / コンポーネント構成を把握できる
- SO アセットの内容を把握できる
- → **コード以外が原因のバグの特定が桁違いに速い**

例えば「シーンに置いた A という GameObject が動かない」というバグでも、
コードを見る前にYAMLを見て `m_IsActive: 0` → 「非アクティブで保存されてた」と
即特定できたり、`{fileID: 0}` → 「参照が外れてる」と発見できたりする。

これは Google AI Studio では絶対に不可能だった。

### Unity MCP（本作では未導入）

Unity Editor との直接連携 MCP（Model Context Protocol）サーバーは存在を把握している程度。
**本作では導入していない**。次作以降で必要があれば検討する。

## AIの弱点を補う仕組み

AIに任せる範囲を広げると、AIの弱点が事故になる。それを防ぐ仕組みを構築した。

### 1. `~/.claude/lessons/` — 教訓ベースの組織知（メイン武器）

これが本作の **AI駆動開発のいちばんの肝**。

#### 仕組み
- 機能実装中・バグ調査中に「**この罠は次回避けたい**」と思ったらマークしておく
- セッション終了時に `/session-close` slash command を実行
- 教訓が `~/.claude/lessons/<category>/LES-XXX-<slug>.md` に記録される
- 次のセッション開始時、SessionStart hook が **[原則]タグの教訓を自動表示**
- AI が「過去の自分の失敗を踏まえた状態」でセッションに入れる

#### カテゴリ実例
- `unity-vr/` — VR固有
- `unity-ui/` — Unity UI レイアウト・SSOT
- `unity-animation/` — Animator 関連
- `unity-performance/` — DrawCall・FFR
- `unity-general/` — Unity 全般
- `debugging/` — デバッグ手法

### 2. lesson 例：スライダー値リセット問題（教訓レコード #011 / 教訓レコード #030 関連）

実例として、何度もハマった「スライダー値が外部からの変更で踏み潰される」問題：

#### 状況
プリセット復元で `parameterValue = 0.8` を state に書き込んだのに、
**次のフレーム**で UI 上のスライダーの古い値（0.5）に戻ってしまう。

#### 原因
`Page_Parameters.Update()` 内で

```csharp
_manager.sensitivity = sensitivitySlider.value;  // 毎フレーム UI → state
```

のように UI 値を state に書き戻す実装が残っていた。
**教訓レコード #011（UIはゲーム状態を映す窓、ゲーム状態がSSOT）違反**。

UI のスライダーは「ゲーム状態の表示」に徹し、状態を**書き戻してはいけない**。

#### 教訓
- 初期表示は `slider.SetValueWithoutNotify(state.value)` で同期
- 値変更は `slider.onValueChanged += v => state.SetValue(v)` で1方向に流す
- `Update()` 内で `state.value = slider.value` という書き戻しを禁止

これを 教訓レコード #011 として記録し、以降の Slider 実装はすべてこのパターンに統一。
**ハマったら教訓化、教訓があれば次は踏まない**サイクル。

### 3. 「環境のせい」逃げの禁止 — 教訓レコード #014

#### 状況（リリース直後の事件）
SteamVR で起動すると XR 初期化失敗で起動しないバグが発生。

#### AI の最初の反応
「OpenXR ランタイムの環境設定の問題かもしれません」「SteamVR のバージョン依存」
など**環境論で逃げよう**としていた。

#### ユーザー側の指摘
「**他の VR 作品は同じ PC・同じ SteamVR 環境で動く**。この作品だけ動かないなら、
プロジェクト側に原因があるはず」

#### 真相
ビルド成果物の `.exe` ファイル名に**非ASCII文字（日本語）**が含まれていたため、
SteamVR の OpenXR ランタイムが exe を見つけられず接続失敗していた。
Product Name を ASCII に変更して解決（教訓レコード #012 として記録）。

#### 教訓
AI の言葉を盲信しない。
比較対象（同じ環境で動いている別作品）があるなら、それと差分を取る。
教訓レコード #013 とセットで「**動くものとの比較解析を最初にやれ**」という原則になった。

### 4. 困ったら**外部リサーチ優先**

AIの内部知識（学習時点までの情報）には限界がある。本作では明示的に：

> AI が解決できないバグや詰まりに遭遇したら、AI 内部の推測に頼らず、
> **Unity の公式ドキュメント・GitHub Issues・Stack Overflow** を WebFetch で
> 当たることを優先する

というルールを CLAUDE.md に書いた。
詳しくは `~/.claude/CLAUDE.md` のデバッグ時原則参照。

### 5. CLAUDE.md の3階層 + 細則ファイル群

`CLAUDE.md` を3階層で配置し、加えて細則ファイル群をプロジェクト配下に置く：

```
~/.claude/CLAUDE.md                ← グローバル方針（全プロジェクト共通）
[Unity Root]/CLAUDE.md             ← Unity 共通ルール
[Project Root]/CLAUDE.md           ← 本プロジェクト固有ルール
[Project]/.claude/rules/*.md       ← UI / VR Interaction / Animation 等の細則ファイル群
```

ただし**本作で機能した順位は**：
1. `~/.claude/lessons/` （最重要・実運用のメイン武器）
2. `[Project]/.claude/rules/*.md` （細則）
3. `[Project]/CLAUDE.md` （プロジェクト固有）
4. `~/.claude/CLAUDE.md` （グローバル）

教訓ファイルが**実運用の主役**で、CLAUDE.md 群は補助。

### 6. カスタムコマンド (skills) — 8本

- **`/investigate`** — バグ・不可解な動作の調査。証拠ベースで原因を絞り込む
- **`/fix-focused`** — 修正指示後の焦点修正。スコープを広げない
- **`/plan-sdd`** — 仕様駆動開発の段階的計画
- **`/implement-phase`** — 計画フェーズの実装
- **`/check-lessons`** — 関連カテゴリ・キーワードで過去の教訓を検索
- **`/convert-to-lessons`** — セッション内の発見を教訓ファイル化
- **`/session-close`** — セッション終了時に教訓を記録 + プロジェクト状態整理
- **`/plan-challenge`** — 計画案の弱点を AI に逆検証させる

これらは AI への指示を**型化**したもの。「指示の質」が「成果物の質」を左右する
LLM の特性を踏まえた最適化。

## 独学者向け学習論：トップダウン × なぞり学習

これは個人の学習スタイルとして見つけた発見。

### 最初に試して失敗したやり方
「ちゃんと自前で理解しながら構築する」

- 1機能ごとに Unity ドキュメントを熟読
- 設計判断を1つずつ自分で下す
- AIには「分からない箇所だけ」聞く
- → 進捗が遅すぎて、学習意欲も折れた

### 切り替えたやり方
「AIに作らせて、できた機能を後ろからなぞって理解する」

1. **AIに目的だけ渡して作らせる**（VRでスライダーで明るさ調節したい等）
2. **動かしてみる**（VR起動）
3. 動いたら、**コードを後ろからなぞる**
4. 「なるほど、`LightingManager.BrightnessMultiplier` プロパティで制御してるのか」
5. 全体像と Unity の流儀が**自然に身についていく**

これは独学者にこそ向いている。AI が全体像を作ってくれるからこそ、
**最大負荷である「ゼロイチ」を高速で超えることができる**。
1人で全部書こうとしたら、形になる前に挫折する。

### この学習論の限界
- **理解の深さが浅い**箇所が残る（読めるけど書けない状態）
- アーキテクチャ判断がAIの「無難な選択」に流れる傾向
- → 経験を積むほど、初手から自分で判断する範囲が広がっていく

それでも、**初リリース作品のスタートとしてはこのやり方が正解**だった、と現在は確信している。

## 既知の運用上の弱点

本ワークフローの限界・自覚している弱点も書いておく。

- **コードレビューの質**：コード全体を毎回精読しないため、設計判断の誤りを見落とすリスクあり。本作の「Page_Environment.cs 1500+行肥大」もこれに起因
- **テスト不足**：ユニットテストが整備されていない。VR実プレイ確認に依存している
- **AIへの依存度**：自分1人で同規模を書ける自信は、現時点ではない
- **設計の一貫性**：スタイルがセッションをまたぐと崩れることがある（LES と CLAUDE.md で抑制しているが完全ではない）

これらは**初リリース作品だから許される**範囲だと思っているが、
業務開発でこの運用をするなら別のリスク対策が必要、と認識している。

## 結論

### このプロジェクトをAI駆動でやってみて分かったこと

1. **プロのような厳密な仕様指示が出せなくても、目的だけAIに伝えて作らせる方が早い**
   - 完璧な仕様書を書こうとせず、目的→生成→違和感→修正のループを回す
2. **メイン役割は指揮者ではなくデバッガー**
   - VR実プレイで違和感を発見し言語化することが人間の核心
3. **AIの「動くはず」「環境のせい」は疑う**
   - 比較解析（動くものとの差分）と外部リサーチ優先で詰める
4. **教訓ベース（lessons）が組織知の代替**
   - 1人開発でも「次の自分」「次のセッションのAI」のために書く
5. **独学はトップダウンから入れ**
   - AIが全体を作ってくれるからこそ、1人でも形にできる
   - 学習の深掘りは「なぞり学習」で後追いで OK

### 今後への展望

- 次作以降で Unity MCP 等の Editor 連携 MCP も検討
- ユニットテスト導入（現在は実プレイ確認のみ）
- アーキテクチャ判断を**AI生成前に自分で詰める**比率を上げる
